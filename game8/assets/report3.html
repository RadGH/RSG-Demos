I

Sprint Overview

What changed across Milestones 63–64

This sprint executed the full roadmap laid out in Report 002. Three bug-fix categories, three new gameplay features, and seven UX improvements were implemented across two milestones. The game is meaningfully richer: players now have gear to choose at creation, camp is more contextual, skill check outcomes show the dice, conditions count down visibly, and the journal distinguishes personal thought from encounter record.

The foundation continues to strengthen — but new code means new edge cases. This report catalogs the sprint's deliverables, surfaces new suspected defects, and sets the next priority list.

III

Features Implemented

New gameplay systems added in Milestones 63–64
FEAT-01 Starting Equipment Choice in Character Creation New
src/screens/CharacterCreation/CharacterCreation.jsx + CSS — Each character now picks one starting item from 3 thematic options based on background archetype (fighter, wilderness, academic, criminal, default). Options are rendered as tappable cards with amber highlight on selection. On Begin Journey, the chosen item is dispatched as ADD_ITEM alongside the default Traveler's Knife.
FEAT-02 Wit < 5 Consequences New
PointBuyPanel.jsx + EncounterScreen.jsx + OutcomeDisplay.jsx — Low-Wit characters now have two effects: (1) A warning in character creation when Wit drops below 5: "⚠ Low wit — may cloud judgment." (2) A 30% chance of a confusion note on skill check outcomes: "[Name] seems confused by what just unfolded." The chance uses the roll value mod 10 for seeded reproducibility when available.
FEAT-03 Camp Scavenge Submenu New
src/screens/Camp/CampScreen.jsx + CSS — The single Scavenge button now expands into 3 options: Search Supplies (original logic), Forage for Food (Nature skill check TN 8, +2–3 food on success, +1 on fail), and Hunt (70% yields +2 food; 30% triggers a "dangerous tracks" message and +1 food). Each option marks the camp-scavenged world flag to prevent infinite repeats. Submenu collapses after any choice.

V

New Suspected Defects

Issues introduced or revealed by this sprint's changes
New code brings new edge cases. None of the following are confirmed — they are risk areas identified through code review of the sprint's changes.
N-001 Starting gear ADD_ITEM dispatches may reference nonexistent CATALOG IDs Risk
CharacterCreation.jsx — The gear lookup table hardcodes item IDs (e.g. 'healing_poultice', 'hand_axe'). If those IDs don't exist in src/data/items.js's CATALOG export, the ADD_ITEM dispatch silently fails and the character receives no gear. Verify all hardcoded IDs against the actual CATALOG object.
N-002 Hunt "combat" path was simplified to avoid triggering real combat — misleading flavor text Risk
CampScreen.jsx — On the 30% dangerous-hunt roll, the implementation shows "You flush out something dangerous… +1 food" without triggering actual combat. The notification implies danger but the outcome is just food. Players may notice the disconnect. Either wire up a real combat trigger or rewrite the flavor text to be honest ("You find tracks but circle back safely").
N-003 Dice roll display only shows when outcome has rolls.roll — many outcomes may never show it Risk
OutcomeDisplay.jsx — The UX-1 roll line only renders when rolls?.roll and rolls?.tn exist. If the encounter engine stores roll data under a different key, or if non-skill-check outcomes don't include it, players will rarely see dice rolls. Audit encounterEngine.js to confirm the exact payload shape and add the fields if missing.
N-004 Weather effects use state.weather — path may not match actual state structure Risk
src/engine/travelEngine.js — The weather block reads state.weather directly. If weather is nested at state.journey.weather or state.world.weather, the condition checks always evaluate to false and weather effects never fire. Verify the correct path in initialState.js.
N-005 Journal fallen-character detection reads party.members which may not be in JournalOverlay context Risk
src/screens/Journal/JournalOverlay.jsx — The dead-author detection looks up entry.characterId in the party members array from state. If JournalOverlay doesn't have access to state.party (depends on what's passed as props vs. context), the lookup returns undefined and no entries are flagged as fallen. Test with a character who has died.
N-006 Forage for Food camp action uses first party member's HED+Nature — doesn't find the best character Minor
src/screens/Camp/CampScreen.jsx — The Forage action description says it uses the "party member with highest Nature skill" but the implementation may default to the first alive character if the skill comparison logic is incomplete. Verify the character selection logic actually finds the max-Nature member.
VI

Still Outstanding from Report 002

Items not yet addressed — carried forward
⚡ Carry Forward
Death deserves a full-screen moment
Character death is still a dismissible toast. A brief black card with the character name and epitaph before the game continues would make death land emotionally. Still the highest-impact UX change not yet made.
⚡ Carry Forward
Combat screen has no stat indicators
Players invest points in Edge, Grit, etc. but combat shows no indication of which stat governs attack or defense. "ATK: Edge+Combat" labels on each character card would make stat investment feel visible and meaningful.
▲ Carry Forward
Combat encounter frequency too low (~10–15%)
Players can march 8–12 days without a fight. Target is 25%. Need 2–3 combat-likely encounters per biome, especially ambush/bandit archetypes. Late-game biomes (Mountains, Storm Coast) are most underserved.
▲ Carry Forward
B-002: Wit < 5 easter egg still unverified end-to-end
The consequences are now partially implemented but the About panel copy still says "Wit below 5 has… consequences" — which implies something more dramatic than a 30% confusion note. Consider amplifying the effect or updating the copy to match.
◆ Carry Forward
Scenario selector not wired to New Game Setup
src/data/scenarios.js is fully defined but scenarios can't be chosen during new game creation. Even one active scenario would add meaningful replayability and validate the scaffolded system.
◆ Carry Forward
Far Shore / victory screen not verified
EndingScreen.jsx exists but it's unclear if the Far Shore biome triggers it correctly or if the journey loops. The victory screen is the emotional payoff of the entire game — needs verification and enrichment.
