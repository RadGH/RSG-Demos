1. Skill scaling audit — the overrun problem

A bug in _mergeInto (skills.js:2149) stacked upgrade damageMult values additively instead of replacing. Designers wrote upgrade descriptions like "damage increases to 200%", but the code did base + 1.5 + 2.0 = 4.6× instead of the intended 2.0×. Talents were correctly additive (they're written as "+30%"); only upgrades needed the semantic swap.

Every bar exceeds the 1× design-intent line by ≥2×. Bone Spike was a 3× outlier because both its L5 and L10 upgrades stacked their full multipliers on top of the base. Full table of all 43 outliers: memory/skill-stacking-audit.md.

2. Balance simulator sweep — finding the 10-round target

After fixing the damage bug, sim parties needed an enemy-HP adjustment to land on the "~10 rounds feels good" target. We tested the user's hypothesis (+500% HP, +250% dmg) and several alternatives against 5 real save files. Saves tested: Necroman (necro+druid L17), Strormu (stormcaller+sorcerer L13), Warrior (warrior+knight L20), Ylva (knight L19 solo), Zanaver (mage+oracle L17).

Boss fight pacing by profile — 150 seeded runs × 9 boss encounters × 5 parties

The user's +500%/+250% hypothesis (red bar, 3rd column) delivered the target round count but crashed boss win rate to 5% — glass builds died before landing enough hits. Pure HP scaling preserved win rates. M266 + 2.0× HP lands exactly on the 10-round target with win rates preserved. Shipped.

Drill-down hierarchy

Post-combat report structure:

  1. Character — one row per party member, sorted by the active tab's total. Each row shows a proportional bar under the name.
  2. Source — skills / abilities / heal kinds / mitigation buckets. Shows hit count, total, and percent of the character's total.
  3. Individual hit — round number, target, actual value, and flags (CRIT, overkill, overheal, mitigation bleed-through).

3b. Meter & Report — M268 updates

After the M267 meter shipped, three follow-ups landed based on playtest feedback.

Combat Report on defeat + final-boss victory

Originally the Combat Report button only appeared on mid-game victory modals. Now on all three end states: mid-game victory, final-boss victory (alongside New Game+), and defeat. The defeat path also gained a View Combat Log button so post-mortem diagnosis doesn't require surviving.

Enemy tracking — opt-in via "Show enemies" checkbox

The meter and report now lazy-create rows for enemies the first time they deal damage, heal, or get mitigated. By default only ally rows show (to preserve the existing signal density). Toggle appears in two places:

  • In-combat meter: a small E checkbox in the panel header. Toggle mid-fight; state doesn't persist across fights.
  • Combat Report overlay: a Show enemies checkbox top-right. Toggling re-renders all three tabs.

Enemy rows render with a purple tint on the proportional bar and a small ENEMY pill before the name, so at a glance you can tell allies from enemies. Useful for "how much incoming damage was the boss soaking vs. trash" post-mortems.

Skill-name attribution fix

All skill hits now carry breakdown.skillName = skill.name, so Magic Missile, Fireball, Shield Bash, Clockwork Bolt, etc. each get their own source row in the report. Previously skill damage was bucketed under "Attack" because the breakdown was only populated by the basic-attack path. The only remaining "Attack" source is actual basic-attack (right-click / no-MP fallback) damage.

Current known gap (flagged in M271 code review)

Enemy heal and mitigation events are still under-reported. Only _meterAddDamage lazy-creates enemy entries via _meterEnsureEntry — the three sibling helpers (_meterAddHeal, _meterAddMit, _meterAddDodge) silently drop events for actors without an existing row. Fix is a 4-line change applied to all three helpers. Flagged in the wishlist.

Because ranks are keyed by nodeId (not slot index), 1 rank = 2 effective ranks. Balance implication: pyromancer and stormcaller are stronger than the audit sim assumed. If you enable "Show enemies" and see enemy damage but little/no enemy heal or mit, that's the bug. 4-line fix flagged on wishlist.
