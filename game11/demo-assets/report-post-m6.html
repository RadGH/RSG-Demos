What Shipped

M2 Polish & Feel
  • Tornado AI rebuilt: 20% direct chase / 40% drift / 40% random with velocity momentum (lerp 0.04) and linger pauses 0.5–2s
  • Zombie edge-respawn every 8 seconds, capped at 12 active
  • Web Audio procedural engine: proximity wind roar, howl overtone, zombie groans, hit SFX, key chime, escape fanfare, ambient storm rumble
  • Visible sprint button (44px circle, bottom-right) — meets touch target spec
  • Building depth shading: south/east shadows, north highlight, 30% lit windows
  • Mini-map: 80×80 overlay rendering player, Sam, helipad, and tornado dots
M3 Map Variety & Survivors
  • Random seed per run — no two layouts identical
  • 3 bonus survivor archetypes: Speed (cyan, +30 px/s), Stamina (orange, 60% drain), Shield (blue, absorbs one hit)
  • Zombie count escalation: 12 baseline → 18 at 90 s → 24 at 150 s
  • Top-5 escape time leaderboard displayed on WinScene
  • First-play tutorial overlay (dismissable, only shown once)
M4 Atmosphere
  • Rain overlay: 80 drops at scrollFactor 0, intensity ramps 0→1 over 120 s
  • Emergency radio broadcast text overlays fire every 25–35 s
  • Panic vignette: concentric border strips with heartbeat pulse active above 50% panic level
  • Tornado trail: 20-position fading dot history rendered at depth 18
M5 Advanced Mechanics
  • Car.js: Arcade physics container (48×28), tap-to-enter/exit, 320 px/s cruise, 448 px/s sprint with velAngle lerp steering
  • 2 cars spawned at road intersections, collide with buildings and zombies
  • Building destruction: tornado proximity drains HP (dt×0.4), static body removed, rubble drawn — deferred mutation queue prevents mid-loop corruption
  • Road flooding: activates at 100 s, 3–5 seed tiles spread 2 tiles every 20 s, applies 0.6× velocity penalty to player
M6 Endgame & Accessibility
  • Score formula: baseScore (300−time) + survivorBonus (150×rescued) + healthBonus (100×HP)
  • Top-5 score leaderboard alongside existing time leaderboard on WinScene
  • Endless mode: title-screen toggle, escalates to 8 tornadoes, no escape condition — death shows "SURVIVAL RECORD"
  • Death recap panel: time survived, survivors rescued count, cause-specific text with tornado coordinates
  • Daily challenge via URL hash #seed=NNNN with SHARE RUN clipboard button on WinScene
  • Accessibility: role="application", aria-live region, _announceStatus() for screen reader events throughout game

Bug Tracker

Known Issues at M6 Close

HIGH AudioContext does not resume after an iOS tab switch — no onstatechange handler wired up. Returning to the game produces silence until refresh.
HIGH Tutorial overlay blocks player input but the enemy and tornado update loop continues advancing — the world progresses while the player is frozen, causing unfair damage on tutorial dismiss.
MED Car container uses axis-aligned Arcade physics body — diagonal wall collisions produce a visually incorrect sliding response rather than a proper deflection.
MED Building destruction index lookup can shift when buildings are spliced — currently safe because destroyed=true short-circuits, but the assumption is fragile and will break if iteration order changes.
MED Flood tile checks only test the player's center tile, not the full player footprint — players near tile edges see inconsistent speed penalties that cut in and out abruptly.
MED Zombie hit-stun does not apply to zombies spawned via the edge-respawn path — overlap wiring differs from initial spawn, so respawned zombies are immune to knockback.
LOW Web Audio wind howl gain is uncapped — can produce a harsh, clipping tone on some speakers or headphones at high panic levels.
LOW Sprint button sets externalSprint directly on InputManager rather than dispatching through an event — tightly coupled and hard to mock in tests.
LOW Zombie groan timers are not staggered at initial spawn — all 12 zombies groan in near-perfect synchrony for the first few seconds, which sounds artificial.
LOW Rain drop positions accumulate a small x-drift over time due to non-wrapped rotation — imperceptible in short sessions but visible after 3+ minutes of gameplay.

Team Retrospective

Developer Feedback

AC
Alex Chen Lead Developer  ·  Game Systems

M2 through M4 were clean. The procedural Web Audio system came together faster than I expected — building each sound as a composited oscillator graph rather than loading audio files was absolutely the right call for mobile, and it shows in the bundle size. The tornado AI behavior model (20/40/40 split with momentum) creates genuinely unpredictable chase patterns that feel alive rather than scripted. That was a good design decision.

"The deferred mutation queue for building destruction was the single best architectural call of the entire block. It cost half a day but saved us from ghost-body crashes that would have taken days to debug."

M5 is where scope got heavy. The car system — Car.js with an Arcade Container, tap-to-enter/exit, and steering lerp — works, but the axis-aligned physics body is a known compromise. The road flooding tile-check using only the player center point is technically wrong. Both of these feel like tech debt that will compound if cars or flooding become more central to the M7+ design.

M6 delivered cleanly. The accessibility work (aria-live, _announceStatus()) was non-trivial to retrofit and I'm glad it's done now rather than at the end of the project. The daily challenge seed implementation is elegant — URL hash with no backend dependency.

What I'd do differently: the sprint button should have gone through the event system from day one. Direct property assignment on InputManager made sense at M2 but it's now a smell that touches every place we add a new input. Event decoupling should be M7 week one, before any new systems land on top of it.

PM Feedback

JR
Jordan Rivera Project Manager  ·  Scope & Delivery

M2 and M3 both shipped on schedule with zero scope cuts. M4 (Atmosphere) was the smoothest milestone of the block — tight scope, high-impact output, and the rain/vignette/radio overlay combination meaningfully elevated the game's tone without adding systemic complexity.

M5 was the heaviest milestone by feature count and it shows. Cars, building destruction, and road flooding are three distinct physics-adjacent systems, each with edge cases. We shipped all three, which is commendable, but the resulting bug density is higher in M5 features than in any other milestone. The flood footprint bug and the car diagonal collision issue are direct consequences of moving fast through complex territory. Next block, I'd recommend breaking anything car/physics-adjacent into its own milestone rather than bundling it with two other systems.

"Six features deferred at M6 close is acceptable — all are polish or optimization items, none are load-bearing for the M7 content plan. Scope discipline held."

The AudioContext/iOS bug is the only item that I would flag as a player-visible defect rather than a developer concern. It needs a fix before any public-facing release. Everything else on the bug list can ship to playtesters without significant impact.

Overall: the block delivered its stated goals — a survivable, atmospheric, replayable mobile game loop with accessibility foundations. We are in a strong position to start M7 with a clear picture of what's owed.

Takeaways

Lessons Learned

1
Deferred mutation queues are mandatory for loop-safe destruction

Removing Arcade physics bodies or splicing arrays mid-update loop causes silent crashes. Use a pending-removal queue flushed after the physics step. This pattern should be standardized and documented for M7+ as more destructible entities are added.

2
Procedural audio is better than audio files for this project scope

Web Audio API oscillator graphs cost zero bytes in the bundle and sound good enough for atmospheric effects. The tradeoff is code complexity — the howl overtone is already unwieldy. Consider extracting audio into a dedicated AudioManager class before adding more sounds in M7.

3
Event-driven input is worth the upfront cost

The sprint button's direct property assignment on InputManager was a one-line shortcut at M2 that is now a pattern inconsistency affecting every new input binding. Event emitters or observer patterns for input state cost a day at the start and save compounding friction across every subsequent milestone.

4
Physics collider assumptions should be stated explicitly in code comments

The car container's axis-aligned body limitation is understood by the developer who wrote it and invisible to everyone else. A one-line comment stating the known limitation and its consequence (diagonal collisions are approximate) would make this acceptable tech debt rather than a confusing bug report.

5
iOS AudioContext lifecycle must be handled from day one

The WebAudio autoplay policy and tab-switch suspension are iOS-specific and not obvious from the Web Audio API docs. Adding an onstatechange resume handler and a user-gesture unlock on first touch should be part of any audio system initialization, not a retrofit. This is now the highest-priority bug before any public release.

6
Mini-map and tornado trail are the dominant per-frame Graphics costs

The mini-map redraws all dots every frame including the static building layer. Caching the building layer as a render texture and only redrawing dynamic dots (player, Sam, helipad, tornado) would cut mini-map cost by roughly 80%. This is straightforward Phaser 3 optimization and should be part of M7 performance pass.

7
Accessibility retrofit is significantly harder than accessibility-first

Adding aria-live regions and _announceStatus() calls across a Phaser 3 canvas game in M6 required touching 9 different files. If aria-live had been wired up at M1, adding status announcements would have been a single function call throughout the project. For the M7 block, any new game state transition should go through the announcer by default.
