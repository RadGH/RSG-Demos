Known Bugs & Issues

Critical
Tornado AI too aggressive — beelines directly to player at 55% rate. Real tornadoes meander unpredictably; current behavior is readable but unfair and kills mid-game tension.
Critical
No zombie respawn — once the initial 12 zombies scatter or are led off-map, they never return. The second half of every run becomes trivially easy. Game balance collapses around the 90-second mark.
Major
Sprint is an invisible tap zone — right-half of screen triggers sprint with no visual affordance. Players don't discover it, and those who do accidentally trigger it. Must be replaced with a visible button in M2.
Major
UIScene tightly coupled to GameScene — UIScene polls GameScene properties every frame. Any refactor of GameScene risks breaking the HUD. Event emitter decoupling is non-negotiable before M3 adds more state.
Major
Suction force stacks without cap — with 4 tornadoes active simultaneously, combined suction vectors can launch the player across the map instantly. No force accumulation cap exists.
Major
No sound effects — complete audio absence. No wind, no zombie groans, no helicopter. Audio is not cosmetic in a survival horror game; its absence significantly reduces tension and immersion.
Minor
Sam beacon edge-clips buildings — occasionally the beacon spawn point lands partially inside a building boundary, making Sam unreachable without deliberate positioning.

UX Review

Mobile Controls

The virtual joystick responds well and hit registration feels accurate. Dead zone tuning is solid. The invisible sprint tap zone is the single biggest UX failure — players cannot be expected to discover it organically. A 44×44px visible button in the lower-right is the fix.

HUD Readability

Health bar and Sam indicator are legible on iPhone 14 Pro. At 393px width the HUD can feel crowded with the mini-map added in M2. Consider anchoring mini-map to top-right and keeping health bottom-left. Sprint button should anchor bottom-right without overlapping joystick dead zone.

Player Orientation

The procedural map provides no consistent landmark. Players lose their position quickly when fleeing tornadoes. The M2 mini-map is essential. Also consider a subtle compass or cardinal direction indicator for "helicopter is north." Right now players backtrack without knowing it.

Tension Curve

First 60 seconds: excellent tension — player is discovering the map while tornado closes in. Minutes 1.5–3: tension drops sharply once zombies scatter. The cliff at the end feels abrupt. Zombie respawn (M2) and zombie scaling (M3) are the primary tension curve fixes.

Accessibility Gaps

No ARIA labels on any interactive elements. Touch targets for buttons meet 44px but margins are tight. No reduced-motion support — tornado animations will cause issues for vestibular-sensitive users. No high-contrast mode. Full pass deferred to M6 but ARIA labels should begin in M2.

First-Play Experience

Zero onboarding. Players open the game and are immediately chased by a tornado with no context. A brief "TORNADO INCOMING — FIND SAM — REACH HELICOPTER" card before play begins would dramatically improve first-run clarity. M3 tutorial overlay is essential, but even a 3-second splash would help in M2.

Game Developer Feedback

Alex Chen

Lead Game Developer  ·  8 years Phaser / game loop engineering

  • Tornado AI must earn unpredictability. Real F3–F4 tornadoes do track warm air and low pressure, so a player-tracking bias is technically defensible — but the movement needs meandering. Implement a wandering offset that rotates over time using a sine wave on the steering angle. A 30% track / 70% wander split will feel natural and genuinely threatening. Erratic is scarier than predictable.
  • Zombie respawn is not optional for balance. "The 3-minute game becomes trivial once zombies scatter." Spawn new zombies at map edge waypoints every 20–30 seconds with a brief telegraph (shadow + footstep sound). Cap total simultaneous count at 8 to manage performance. Never let the map feel empty.
  • Sound is 50% of the horror. "We need at least procedural wind before M3." Web Audio API can generate convincing low-frequency wind rumble with a BiquadFilterNode on noise — no audio files needed. A looping wind node with gain mapped to nearest tornado distance will transform the game feel overnight. This should not wait until M4.
  • Performance risk: debris + 4 tornadoes + 12 zombies on low-end devices. Each Phaser particle emitter is cheap in isolation; 4 heavy emitters simultaneously is not. Pool particles aggressively. Consider LOD for tornado visuals at range — full spiral when close, simpler rotating sprite when distant. Profile on a 2021-era mid-range Android before M2 ships.
  • UIScene coupling must die before M3. Any feature that adds game state — survivors, panic meter, car health — will require UIScene to know about it. Switch to Phaser's built-in event emitter: this.events.emit('healthChanged', value) in GameScene, this.scene.get('GameScene').events.on('healthChanged', ...) in UIScene. One-day refactor that saves weeks of pain.
  • Building destruction (M5) is the feature I'm most excited about. Watching a building collapse in the tornado's path as you run past it is the defining cinematic moment for this game. Plan it carefully — use a multi-frame sprite sheet or Phaser tweens on building alpha/scale with debris particles. The challenge is making it feel weighty without destroying frame time.

Project Manager Feedback

Jordan Rivera

Project Manager  ·  Indie game production, scope & delivery

  • M2 scope is dense — watch for scope creep on sound. Audio implementation tends to balloon. If sound design runs long, split into M2 (basic SFX via Web Audio) and M2b (polish and proximity effects). Do not let audio block zombie respawn, mini-map, or the sprint button fix. Those three items are the M2 must-haves.
  • Prioritize the tornado fix first in M2. It's the highest-impact single change in the entire roadmap. A game where the tornado feels threatening and unpredictable is a fundamentally better game. Fix it on day one of M2 so every subsequent playtest builds on improved baseline feel.
  • Random seeds (M3) are essential before social features. Do not add a leaderboard, daily challenge, or any shareable feature until every run generates a unique map. Veterans memorizing the fixed seed layout will undermine competitive features entirely. Seed-first is non-negotiable.
  • Building destruction (M5) is high-risk — plan as a stretch goal. Scoping it as optional gives M5 a safe fallback: if destruction proves too complex or performance-costly, ship M5 without it and carry it to M6. Do not block the car mechanic or panic meter on destruction being complete.
  • Leaderboard in M3 is low-effort, high-reward. LocalStorage top-5 is a single afternoon of work and dramatically increases session replay. Players who finish in 2:10 will immediately attempt 1:58. Add it early and let it sit — no maintenance burden.
  • Overall timeline: M2–M6 in 6 sessions if scoped correctly. M2 and M3 are the heavy ones (2 sessions each). M4 and M5 are feature-dense but well-defined. M6 is polish and should be treated as a full QA/accessibility sprint, not another feature milestone. Protect M6 scope ruthlessly.

Limitations & Risks

🔇
No audio system exists. Web Audio API requires explicit user gesture to initialize in mobile Safari. The AudioContext must be created or resumed on first touch. Failing to handle this means all audio is silently skipped on iOS with no error — extremely easy to miss during desktop testing.
⚡
Suction force stacking can break physics. Four tornadoes within range simultaneously apply additive force vectors with no accumulation cap. At 4× suction, velocity can hit values that skip collision detection entirely (tunneling). Add a hard max velocity clamp and per-frame force cap before M3 adds more tornadoes.
🗺️
Phaser Graphics map won't scale to tilemap performance. The current procedural map uses Graphics objects which re-render on every draw call. This is fine for one map, but adding destruction (M5) means dynamic Graphics updates every frame. Evaluate whether a static RenderTexture bake per map section would improve perf before M5 begins.
🔁
Fixed seed ruins replayability and competitive features. Every session is identical: same map, same tornado spawns, same Sam position. Veterans beat the game in under 90 seconds once memorized. Random seeds must land in M3 before any social or competitive layer is built.
💾
No persistence beyond best time. M3 leaderboard is the only planned persistence before M6. Settings, accessibility preferences, and control options have no save mechanism. If M4 adds audio volume sliders, we need a localStorage settings object. Plan this infrastructure in M3 even if only partially used.
📱
Low-end mobile performance is an unknown. Particle debris + 4 tornadoes + 12 zombies + rain overlay (M4) is a lot of simultaneous rendering. A Pixel 4a or similar 2019-era device should be the minimum viable target. No performance profiling has been done at maximum entity count. This needs a dedicated profiling pass in M2.
